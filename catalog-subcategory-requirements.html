Background

Treez maintains a global set of canonical categories and subcategories. These are Treez-owned values that drive purchase limits, tax rules, compliance reporting (Metrc), and Sell Treez subtype resolution. They cannot be freely renamed or removed without downstream consequences.

Dispensaries increasingly need catalog names that reflect how they merchandise — not how Treez internally classifies products. A dispensary selling "Rosin" doesn't want their product dropdown to say "Concentrate — Rosin"; they want "Rosin" or whatever their brand language calls it. Today there is no mechanism to do this without breaking compliance logic.

This feature introduces org-specific custom subcategories: display aliases that sit on top of the global canonical layer. Products retain their canonical subcategory assignment (and all the compliance metadata that comes with it), while the org sees the name they've chosen.

Core Concepts

Global Canonical Subcategories

Treez-owned. Seeded and maintained by Treez. Cannot be deleted or renamed by orgs. Orgs can only control whether a global subcategory is included or excluded from the product create/edit dropdown.

  • Purchase limits
  • Tax rule application
  • Metrc compliance reporting
  • Sell Treez subtype resolution

Org-Specific Custom Subcategories

Display aliases created by an org, mapped 1:1 to a canonical parent.

  • One canonical can have many customs
  • Inherits all compliance metadata from canonical
  • Canonical productSubCategoryId stays source of truth
  • Nullable customSubCategoryId added alongside it

Activate / Deactivate

Orgs control which global subcategories are active for their catalog on a per-category basis. A deactivated subcategory:

  • Does not appear in the product editing dropdown for that org
  • Remains selectable in the automated collections builder in MVP — existing products may still be assigned to it and need to be collectable. Filtering the rule builder to active-only subcategories is fast follow.
  • Does not affect existing product assignments — deactivation only controls forward assignment

MVP Scope MVP

Initial rollout: Story is the first org enabled. The MVP scope below reflects the Story launch. Additional orgs are enabled org by org as the feature flag rolls out. Fast follow capabilities are required before general availability to additional orgs.
Feature flag: Catalog Management - Custom Subcategory Management (managed in organization-management service). The management surface — Manage > Subcategories page, activate/deactivate, and custom subcategory CRUD — is gated behind this flag and enabled per org. Schema changes and ProductCore v1x3x0 event fields ship to all orgs regardless of the flag.

Manage > Subcategories Page

The starting point is a category selector. After choosing a category, the page shows a single unified list of all global subcategories — both active and deactivated — with a Status filter (Active / Inactive / All) in the list toolbar. The default view is Active, which shows active globals and deactivated globals that have custom subcategories under them.

Activate / Deactivate
  • Activate and Deactivate are actions in the kebab menu on each global row.
  • Deactivated globals are visually muted. A blue "Using custom subcategories" badge appears on deactivated globals that have customs — communicating implicitly that the global is not directly assignable but its customs are active.
  • Deactivation only controls forward assignment — existing product assignments on a deactivated global are unaffected.
  • The Global badge distinguishes canonical subcategories from org-created customs.
Deactivation Rules for Global Subcategories
  • No customs, no products: deactivates immediately.
  • No customs, products assigned: confirmation prompt warns that existing product assignments are unaffected but new assignments will be blocked. Operator confirms to proceed.
  • Customs exist, none have products: confirmation prompt fires before cascade-deleting empty customs and deactivating the global.
  • Customs exist with products assigned: blocked. A modal explains which custom subcategories have products assigned and instructs the operator to reassign or delete them first.
Shadowing
  • Creating the first custom subcategory under a global automatically shadows it — the global row becomes visually muted and displays a blue "Using custom subcategories" badge.
  • A shadowed global can no longer be directly assigned to new products; products must be assigned to a custom. Existing product assignments on the global are unaffected.
  • Deleting the last custom subcategory under a global removes the shadow and restores the global to active.
  • Activate is not shown in the kebab for shadowed globals — the only path back to active is deleting all customs.
Custom Subcategories
  • Add: Inline form under the global row. Available on active globals and shadowed globals. Name is required. A custom subcategory may share the same name as its parent global, but duplicate names among other customs under the same global are not allowed.
  • Edit Name: Rename a custom subcategory. Name change only — canonical mapping does not change, no products move. ST reflects the new name after the existing rename event fan-out completes.
  • Delete: Only available when zero products are assigned. If products are assigned, Delete does not appear in the kebab.

Catalog — Product List & Product Card

Product List Grid
  • The subcategory column shows the custom subcategory name when assigned to a custom, otherwise the global name.
Product Card — Create / Edit
  • The subcategory dropdown shows custom subcategory names where they exist under an active global, otherwise the global name. Deactivated global subcategories do not appear in the dropdown.
Product List Filters
  • Subcategory filter options in the catalog product list continue to display all subcategories (global and custom) in MVP — not yet restricted to the org's included set.
  • Filtering by active/deactivated state will be revisited after the reassignment flow is in place, once it is safe to hide options without stranding products in unfiltered states.

Collections

Collection membership evaluation is not changed at MVP. Existing rules continue to evaluate against canonical subcategory assignments. When products are reassigned to a different subcategory, the reassignment flow surfaces a count of automated collections that reference the source subcategory via an inclusion or exclusion rule — informational only, so operators can audit manually. Collection rules are not rewritten automatically.

No Rule Impact from Activate/Deactivate or Custom Subcategory Creation
  • Excluding, including, or creating a custom subcategory does not affect existing collection rules. Rules remain syntactically valid and collection membership is unchanged.
  • Deleting a custom subcategory with zero products assigned leaves any referencing rule entry inert (the deleted UUID returns no products) but does not break the collection. Users can clean up stale rule entries manually.
Product Assignment and Collection Rules
  • Products are assigned to custom subcategories via the catalog product dropdown and API in MVP. When reassigning products to a different subcategory, the reassignment flow shows how many automated collections reference the source subcategory via an inclusion or exclusion rule. This count is informational — collection rules are not rewritten automatically, and operators should audit affected collections manually after reassigning.
Collection Rule Builder
  • The rule builder must expose custom subcategory names, active globals with customs (products may still be assigned there), and deactivated globals (products may still be assigned there too). Showing the full option set with custom names ships as fast follow.

Product Management Read Service (PMRS)

Resolved Subcategory Display Name
  • PMRS exposes a resolved subCategoryDisplayName field on its product search API. The field falls back from the org-specific custom subcategory name to the global subtype, so a single field always carries the right label whether or not the product has a custom subcategory assigned. Downstream consumers read one field — no fallback logic required on the client side.

Fast Follow Fast Follow

The following capabilities ship after MVP, ordered by priority. Items 1–2 are tightly coupled — they share the same reassignment modal surface and PMRS fan-out infrastructure. They are scoped and built together.

1. Reassignment Fan-out Highest priority — coupled infrastructure

On reassignment confirmed, PMS queries all affected products and emits a ProductCore v1x3x0 event per product; PMRS updates subtype per product via existing event processing.

Reassign Products (custom subcategory kebab)

  • "Reassign Products" added to the kebab when products are assigned (Delete remains hidden in this state).
  • Title: "Reassign [Custom Subcategory Name]" — Body: "{n} products will be reassigned and the subcategory will be deleted."
  • "This action cannot be undone." in red. Confirm disabled until target selected. Cannot be skipped.
  • Dropdown: other customs under active globals, or globals with no customs. Source excluded from list. Bulk type subcategories can only be reassigned to other bulk subcategories — non-bulk targets are not shown when the source is bulk, and vice versa.
  • On confirm: products move to target, source custom deleted. Does not remain as an empty shell.
  • Success toast: "Reassignment started. View Job" — "View Job" links to the job status page.
2. Collection Count in Reassign Modal Ships with #1 — same modal surface
  • When the reassign modal opens, a count of automated collections that reference the source subcategory via an inclusion or exclusion rule is fetched from the product-collections service.
  • If n > 0: "{n} automated collection(s) use this subcategory in a rule. Review these collections after reassigning."
  • If fetch fails or is loading, modal remains fully functional — action is not blocked.
  • Collection rules are not rewritten automatically. Operators must audit and update affected collections manually.
4. Catalog-to-ST Reconciliation — Seed & Migrate End-to-end proof of concept — gated on client comms
  • Class 1 canonical renames, Class 2 seeded default customs, Class 3 missing canonicals. See Catalog Reconciliation section for class definitions.
  • Requires client communication before running — orgs need to know product subcategory assignments may change.
  • Validates the full custom subcategory pipeline end-to-end across all orgs: custom subcategory creation, product fan-out, and Sell Treez name display.
5. First-Custom Reassign Prompt Small addition once reassignment exists
  • When creating the first custom under a global that has products assigned, a prompt offers to reassign those products to the new custom now.
  • Only triggered on first custom creation. Skipping is allowed — products can be reassigned later via the kebab.
6. Collections MFE — Filter Rule Options to Org-Active Subcategories Independent work
  • Post activate/deactivate, the collection rule builder must fetch only the org's active subcategories (and their custom subcategories) as rule options.
  • Prevents users from building rules on deactivated subcategories.
7. Product Groups — Legacy ST Discount Management Post-Story MVP
  • The subcategory filter in legacy ST discount management (Product Groups) surfaces custom subcategory names.
  • Rules can be built against any active custom or global subcategory.
8. Import Service — Accept Custom Subcategory by Name or ID API path available in MVP
  • File-based bulk imports accept custom subcategory display name or ID — non-technical self-service path.
  • Bulk re-tagging via catalog API is available in MVP; import adds the self-service path.
9. Custom Subcategory Import Type Lower priority — migration from other POS systems
  • New import type that allows orgs to define their custom subcategory structure in bulk. Columns: Product Category, Custom Product Subcategory, Global Subcategory.
  • Enables orgs migrating from other POS systems to establish their full custom subcategory mapping before importing products.
  • On import, each row creates a custom subcategory (or matches an existing one) under the specified global subcategory for the org.
10. Product Import — Custom Subcategory Column Lower priority — migration from other POS systems
  • Extends the existing product import to accept a custom subcategory column.
  • When a matching custom subcategory exists for the org, the imported product is assigned to it on create or update.
  • Allows orgs to migrate product catalogs from other POS systems with subcategory assignments intact in a single import pass.

Collections Impact

How Collection Rules Reference Subcategories

Collection rules can reference either a custom subcategory or a global canonical subcategory, depending on what the operator selects in the rule builder. A rule targeting a custom subcategory matches only products assigned to that custom. A rule targeting a global canonical matches only products still directly assigned to that canonical — not products assigned to customs under it.

Collection membership evaluation is not changed at MVP. Rules continue to evaluate against the canonical productSubCategoryId. Automatic evaluation updates were rejected because they would silently add or remove products from collections when products are reassigned to a different subcategory. Instead, the reassignment flow surfaces a collection impact count so operators can audit affected rules manually.

Product subcategory assignment can be in any of four states, and the rule builder must expose all of them:

  • Included global (no customs) — the standard case; products are assigned directly to the global
  • Custom subcategory — products explicitly assigned to an org custom
  • Included global (customs exist) — products assigned before customs were created remain on the global until explicitly reassigned
  • Deactivated global — products on an excluded global remain there; exclusion does not migrate existing assignments

Renaming a custom subcategory does not affect which products a rule matches — only the display name changes. Reassigning products to a different subcategory can change which collections they appear in; the reassignment flow shows a count of affected collections so operators can audit rules manually.

Transitional note (MVP): In the window before bulk reassignment ships, products may be split across a global and its customs — existing products remain directly on the global until explicitly reassigned, while new products are assigned to customs. During this period, a rule targeting only CUSTOM1 will miss products still on GLOBAL1. To capture the full product set, include both in the rule. Once bulk reassignment ships and products are moved off the global, this consideration no longer applies.

Reassignment and Collection Impact

When a bulk reassignment is confirmed, collection rules that reference the source subcategory are not automatically rewritten. Auto-rewriting collection rules was rejected because it would silently add or remove products from collections. Instead, the reassignment modal shows a count of collections referencing the source subcategory via an inclusion or exclusion rule, so operators can audit and update those rules manually after reassigning. Collections are not republished automatically on reassignment.

Collections Rule Builder Fast Follow

Today the rule builder shows all global subcategories. Once activate/deactivate is live, it must show only the org's active subcategories and their custom subcategories as options. This prevents operators from accidentally building rules on deactivated subcategories.

Technical Debt — Catalog-to-ST Reconciliation

The current catalog and Sell Treez subtype data have diverged over time. Three classes of reconciliation work are needed before the custom subcategory feature is fully clean. Reconciliation does not block the user-defined custom subcategory feature — the PMRS hardcoded map stays as a fallback throughout. The audit is MVP; the seed + migrate is fast follow, gated on client communication.

Class 1

Name Mismatch — Rename in Place

A subcategory exists in both catalog and ST but the names don't match. Resolution: rename the catalog subcategory to match ST (or the canonical standard). No migration of product assignments needed.

Class 2

Catalog-Only Subcategory — Seed Default Custom

A subcategory exists in the catalog but has no matching ST subtype. Resolution: seed a default custom subcategory for each org using these values. Products remain on the same name, now backed by the nearest canonical. The seeded default can be deleted by the org if no products are assigned.

Class 3

ST Subtype Missing in Catalog — Seed Canonical

A subtype exists in ST but is absent from the global catalog. Resolution: seed the canonical in the catalog so the full chain is consistent.

PMRS

Remove typeToCategorySubTypeMap

PMRS currently maintains a hardcoded map from catalog types to Sell Treez subtypes. After reconciliation, this is eliminated and Sell Treez reads subtype directly from the canonical subcategory ID — a passthrough with no translation layer.

Client communication required before reconciliation runs. The Class 2 migration — seeding default customs and re-tagging existing products across all orgs — is the highest blast-radius action in this effort. Orgs need to be notified that their product subcategory assignments may change before this runs. This gates the seed + migrate fast follow ticket.

Event Model

Product Events

Existing product events gain additive fields for the assigned custom subcategory name and ID. No breaking change to existing consumers. All orgs receive these fields regardless of the feature flag — the fields are simply empty for orgs without custom subcategories.

Custom Subcategory Lifecycle — Create / Update / Delete

Three new events cover the full lifecycle of a custom subcategory. Downstream read services consume these to stay in sync.

  • Created — emitted when an org creates a new custom subcategory. Read services update their records accordingly.
  • Updated (rename) — emitted when a custom subcategory is renamed. Triggers a fan-out to all products assigned to that custom so Sell Treez and other consumers see the updated name. Uses the same fan-out pattern as brand name edits. Ships in MVP.
  • Deleted — emitted after the product management service has already cleared product assignments. Read services remove their records.
Reassignment Event Fast Follow

When products are bulk reassigned from one subcategory to another, a reassignment event is emitted. Collection rules are not auto-rewritten — the reassignment modal surfaces a collection impact count so operators can audit rules manually. Reassignments without an explicit replacement target do not trigger this event.

Note on rename: Rename ships in MVP. The existing fan-out infrastructure handles name propagation to all downstream consumers — no new systems required. Bulk reassignment fan-out ships as fast follow.

Open Questions

Subcategory filter in product search / FOH

Today the subcategory filter in product search shows all options. Post-activate/deactivate, should it filter to only subcategories with products assigned in the org's catalog? Likely yes, but may depend on having auto-reassignment in place first to avoid showing empty filters.

Collections MFE rule builder for custom subcategories

When an org has custom subcategories, should the rule builder show the custom name, the canonical name, or both? The rule stores the canonical ID — the display is a UX decision.

Org-specific display name resolution in the collection product grid

The collection detail grid currently resolves subcategory name from the canonical productSubCategory table. For org-specific names to show in that grid, PMRS/FOH resolution needs to route through the custom subcategory table. Scope TBD.

Seeded default custom subcategories (Class 2 reconciliation)

When seeded defaults are created for an org, should they pre-populate as custom subcategories on the Manage page, or should they appear as globals that the org can optionally customize? Needs design alignment.
